\begin{DoxyPre}<test-\/spec> ...\end{DoxyPre}


\doxylink{class_test}{Test} cases, wildcarded test cases, tags and tag expressions are all passed directly as arguments. Tags are distinguished by being enclosed in square brackets.

If no test specs are supplied then all test cases, except \char`\"{}hidden\char`\"{} tests, are run. \doxylink{struct_a}{A} test is hidden by giving it any tag starting with (or just) a period ({\ttfamily .}) -\/ or, in the deprecated case, tagged {\ttfamily \mbox{[}hide\mbox{]}} or given name starting with `'./\textquotesingle{}{\ttfamily . To specify hidden tests from the command line}{\ttfamily \mbox{[}.\mbox{]}}{\ttfamily or}{\ttfamily \mbox{[}hide\mbox{]}}\`{} can be used {\itshape regardless of how they were declared}.

Specs must be enclosed in quotes if they contain spaces. If they do not contain spaces the quotes are optional.

Wildcards consist of the {\ttfamily $\ast$} character at the beginning and/or end of test case names and can substitute for any number of any characters (including none).

\doxylink{class_test}{Test} specs are case insensitive.

If a spec is prefixed with {\ttfamily exclude\+:} or the {\ttfamily $\sim$} character then the pattern matches an exclusion. This means that tests matching the pattern are excluded from the set -\/ even if a prior inclusion spec included them. Subsequent inclusion specs will take precedence, however. Inclusions and exclusions are evaluated in left-\/to-\/right order.

\doxylink{class_test}{Test} case examples\+:

\DoxyCodeLine{thisTestOnly\ \ \ \ \ \ \ \ \ \ \ \ Matches\ the\ test\ case\ called,\ 'thisTestOnly'}
\DoxyCodeLine{"{}this\ test\ only"{}\ \ \ \ \ \ \ \ Matches\ the\ test\ case\ called,\ 'this\ test\ only'}
\DoxyCodeLine{these*\ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ Matches\ all\ cases\ starting\ with\ 'these'}
\DoxyCodeLine{exclude:notThis\ \ \ \ \ \ \ \ \ Matches\ all\ tests\ except,\ 'notThis'}

Names within square brackets are interpreted as tags. \doxylink{struct_a}{A} series of tags form an AND expression whereas a comma-\/separated sequence forms an OR expression. e.\+g.\+:


\begin{DoxyPre}[one][two],[three]\end{DoxyPre}
 This matches all tests tagged {\ttfamily \mbox{[}one\mbox{]}} and {\ttfamily \mbox{[}two\mbox{]}}, as well as all tests tagged {\ttfamily \mbox{[}three\mbox{]}}

\doxylink{class_test}{Test} names containing special characters, such as {\ttfamily ,} or {\ttfamily \mbox{[}} can specify them on the command line using {\ttfamily \textbackslash{}}. {\ttfamily \textbackslash{}} also escapes itself.

 \hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md56}{}\doxysubsection{Choosing a reporter to use}\label{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md56}

\begin{DoxyPre}-\/r, -\/-\/reporter <reporter[::output-\/file]>\end{DoxyPre}


\texorpdfstring{$>$}{>} Support for providing output-\/file through the {\ttfamily -\/r}, {\ttfamily -\/-\/reporter} flag was \href{https://github.com/catchorg/Catch2/pull/2183}{\texttt{ introduced}} in Catch2 X.\+Y.\+Z

Reporters are how the output from Catch2 (results of assertions, tests, benchmarks and so on) is formatted and written out. The default reporter is called the \char`\"{}\+Console\char`\"{} reporter and is intended to provide relatively verbose and human-\/friendly output.

There are multiple built-\/in reporters, you can see what they do by using the \href{command-line.md\#listing-available-tests-tags-or-reporters}{\texttt{ {\ttfamily -\/-\/list-\/reporter}}} flag. If you need a reporter providing custom format outside of the already provided ones, look at the \href{reporters.md\#writing-your-own-reporter}{\texttt{ "{}write your own reporter"{} part of the reporter documentation}}.

This option may be passed multiple times to use multiple (different) reporters at the same time. See the \href{reporters.md\#multiple-reporters}{\texttt{ reporter documentation}} for details on what the resulting behaviour is. Also note that at most one reporter can be provided without the output-\/file part of reporter spec. This reporter will use the \char`\"{}default\char`\"{} output destination, based on the \doxylink{md_ext_2hera_2extern_2_catch2_2docs_2command-line_sending-output-to-a-file}{\`{}-\/o\`{}, \`{}-\/-\/out\`{}} option.

\begin{quote}
Support for using multiple different reporters at the same time was \href{https://github.com/catchorg/Catch2/pull/2183}{\texttt{ introduced}} in Catch2 X.\+Y.\+Z \end{quote}
As with the {\ttfamily -\/-\/out} option, using {\ttfamily -\/} for the output file name sends the output to stdout.

{\itshape Note\+: There is currently no way to escape {\ttfamily \+::} in the reporter spec, and thus reporter/file names with {\ttfamily \+::} in them will not work properly. As {\ttfamily \+::} in paths is relatively obscure (unlike {\ttfamily \+:}), we do not consider this an issue.}

Under most debuggers Catch2 is capable of automatically breaking on a test failure. This allows the user to see the current state of the test during failure.

\begin{DoxyPre}-\/s, -\/-\/success\end{DoxyPre}


Usually you only want to see reporting for failed tests. Sometimes it\textquotesingle{}s useful to see {\itshape all} the output (especially when you don\textquotesingle{}t trust that that test you just added worked first time!). To see successful, as well as failing, test results just pass this option. Note that each reporter may treat this option differently. The Junit reporter, for example, logs all results regardless.

 \hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md59}{}\doxysubsection{Aborting after a certain number of failures}\label{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md59}

\begin{DoxyPre}-\/a, -\/-\/abort
-\/x, -\/-\/abortx [<failure threshold>]
\end{DoxyPre}


If a {\ttfamily REQUIRE} assertion fails the test case aborts, but subsequent test cases are still run. If a {\ttfamily CHECK} assertion fails even the current test case is not aborted.

Sometimes this results in a flood of failure messages and you\textquotesingle{}d rather just see the first few. Specifying {\ttfamily -\/a} or {\ttfamily -\/-\/abort} on its own will abort the whole test run on the first failed assertion of any kind. Use {\ttfamily -\/x} or {\ttfamily -\/-\/abortx} followed by a number to abort after that number of assertion failures.

\begin{DoxyPre}-\/n, -\/-\/name <name for test run>\end{DoxyPre}


If a name is supplied it will be used by the reporter to provide an overall name for the test run. This can be useful if you are sending to a file, for example, and need to distinguish different test runs -\/ either from different Catch executables or runs of the same executable with different options. If not supplied the name is defaulted to the name of the executable.

\label{md_ext_2hera_2extern_2_catch2_2docs_2command-line_eliding-assertions-expected-to-throw}%

 \hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md63}{}\doxysubsection{Eliding assertions expected to throw}\label{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md63}

\begin{DoxyPre}-\/e, -\/-\/nothrow\end{DoxyPre}


Skips all assertions that test that an exception is thrown, e.\+g. {\ttfamily REQUIRE\+\_\+\+THROWS}.

These can be a nuisance in certain debugging environments that may break when exceptions are thrown (while this is usually optional for handled exceptions, it can be useful to have enabled if you are trying to track down something unexpected).

Sometimes exceptions are expected outside of one of the assertions that tests for them (perhaps thrown and caught within the code-\/under-\/test). The whole test case can be skipped when using {\ttfamily -\/e} by marking it with the {\ttfamily \mbox{[}!throws\mbox{]}} tag.

When running with this option any throw checking assertions are skipped so as not to contribute additional noise. Be careful if this affects the behaviour of subsequent tests.

\begin{DoxyPre}-\/i, -\/-\/invisibles\end{DoxyPre}


If a string comparison fails due to differences in whitespace -\/ especially leading or trailing whitespace -\/ it can be hard to see what\textquotesingle{}s going on. This option transforms tabs and newline characters into {\ttfamily \textbackslash{}t} and {\ttfamily \textbackslash{}n} respectively when printing.

\begin{DoxyPre}-\/w, -\/-\/warn <warning name>\end{DoxyPre}


You can think of Catch2\textquotesingle{}s warnings as the equivalent of {\ttfamily -\/Werror} ({\ttfamily /\+WX}) flag for C++ compilers. You can enable multiple warnings at the same time.

 \hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md66}{}\doxysubsection{Reporting timings}\label{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md66}

\begin{DoxyPre}-\/d, -\/-\/durations <yes/no>\end{DoxyPre}


When set to {\ttfamily yes} Catch will report the duration of each test case, in milliseconds. Note that it does this regardless of whether a test case passes or fails. Note, also, the certain reporters (e.\+g. Junit) always report test case durations regardless of this option being set or not.


\begin{DoxyPre}-\/D, -\/-\/min-\/duration <value>\end{DoxyPre}


\texorpdfstring{$>$}{>} {\ttfamily -\/-\/min-\/duration} was \href{https://github.com/catchorg/Catch2/pull/1910}{\texttt{ introduced}} in Catch2 2.\+13.\+0

When set, Catch will report the duration of each test case that took more than \texorpdfstring{$<$}{<}value\texorpdfstring{$>$}{>} seconds, in milliseconds. This option is overriden by both {\ttfamily -\/d yes} and {\ttfamily -\/d no}, so that either all durations are reported, or none are.

\begin{DoxyPre}-\/-\/order <decl|lex|rand>\end{DoxyPre}


\doxylink{class_test}{Test} cases are ordered one of three ways\+:\hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md69}{}\doxysubsubsection{decl}\label{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md69}
Declaration order (this is the default order if no --order argument is provided). Tests in the same TU are sorted using their declaration orders, different TUs are in an implementation (linking) dependent order.\hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md70}{}\doxysubsubsection{lex}\label{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md70}
Lexicographic order. Tests are sorted by their name, their tags are ignored.\hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md71}{}\doxysubsubsection{rand}\label{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md71}
Randomly sorted. The order is dependent on Catch2\textquotesingle{}s random seed (see \doxylink{md_ext_2hera_2extern_2_catch2_2docs_2command-line_rng-seed}{\`{}-\/-\/rng-\/seed\`{}}), and is subset invariant. What this means is that as long as the random seed is fixed, running only some tests (e.\+g. via tag) does not change their relative order.

\begin{quote}
The subset stability was introduced in Catch2 v2.\+12.\+0 \end{quote}

 \hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md72}{}\doxysubsection{Specify a seed for the Random Number Generator}\label{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md72}

\begin{DoxyPre}-\/-\/rng-\/seed <'time'|'random-\/device'|number>\end{DoxyPre}


Sets the seed for random number generators used by Catch2. These are used e.\+g. to shuffle tests when user asks for tests to be in random order.

Using {\ttfamily time} as the argument asks Catch2 generate the seed through call to {\ttfamily std\+::time(nullptr)}. This provides very weak randomness and multiple runs of the binary can generate the same seed if they are started close to each other.

Using {\ttfamily random-\/device} asks for {\ttfamily std\+::random\+\_\+device} to be used instead. If your implementation provides working {\ttfamily std\+::random\+\_\+device}, it should be preferred to using {\ttfamily time}. Catch2 uses {\ttfamily std\+::random\+\_\+device} by default.

\label{md_ext_2hera_2extern_2_catch2_2docs_2command-line_run-section}%
\Hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2command-line_run-section}%
 \hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md81}{}\doxysubsection{Specify the section to run}\label{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md81}

\begin{DoxyPre}-\/c, -\/-\/section <section name>\end{DoxyPre}


To limit execution to a specific section within a test case, use this option one or more times. To narrow to sub-\/sections use multiple instances, where each subsequent instance specifies a deeper nesting level.

E.\+g. if you have\+:


\begin{DoxyPre}
TEST\_CASE( "{}Test"{} ) \{
  SECTION( "{}sa"{} ) \{
    SECTION( "{}sb"{} ) \{
      /*...*/
    \}
    SECTION( "{}sc"{} ) \{
      /*...*/
    \}
  \}
  SECTION( "{}sd"{} ) \{
    /*...*/
  \}
\}
\end{DoxyPre}


Then you can run {\ttfamily sb} with\+: 
\begin{DoxyPre}./MyExe \doxylink{class_test}{Test} -\/c sa -\/c sb\end{DoxyPre}


Or run just {\ttfamily sd} with\+: 
\begin{DoxyPre}./MyExe \doxylink{class_test}{Test} -\/c sd\end{DoxyPre}


To run all of {\ttfamily sa}, including {\ttfamily sb} and {\ttfamily sc} use\+: 
\begin{DoxyPre}./MyExe \doxylink{class_test}{Test} -\/c sa\end{DoxyPre}


There are some limitations of this feature to be aware of\+:
\begin{DoxyItemize}
\item Code outside of sections being skipped will still be executed -\/ e.\+g. any set-\/up code in the TEST\+\_\+\+CASE before the start of the first section.
\item At time of writing, wildcards are not supported in section names.
\item If you specify a section without narrowing to a test case first then all test cases will be executed (but only matching sections within them).
\end{DoxyItemize}

\begin{DoxyPre}-\/\#, -\/-\/filenames-\/as-\/tags\end{DoxyPre}


When this option is used then every test is given an additional tag which is formed of the unqualified filename it is found in, with any extension stripped, prefixed with the {\ttfamily \#} character.

 \hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md83}{}\doxysubsection{Override output colouring}\label{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md83}

\begin{DoxyPre}-\/-\/use-\/colour <yes|no|auto>\end{DoxyPre}


Catch colours output for terminals, but omits colouring when it detects that output is being sent to a pipe. This is done to avoid interfering with automated processing of output.

{\ttfamily -\/-\/use-\/colour yes} forces coloured output, {\ttfamily -\/-\/use-\/colour no} disables coloured output. The default behaviour is {\ttfamily -\/-\/use-\/colour auto}.

 \hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md84}{}\doxysubsection{Test Sharding}\label{md_ext_2hera_2extern_2_catch2_2docs_2command-line_autotoc_md84}

\begin{DoxyPre}-\/-\/shard-\/count <\#number of shards>, -\/-\/shard-\/index <\#shard index to run>\end{DoxyPre}


\texorpdfstring{$>$}{>} \href{https://github.com/catchorg/Catch2/pull/2257}{\texttt{ Introduced}} in Catch2 X.\+Y.\+Z.

When {\ttfamily -\/-\/shard-\/count \texorpdfstring{$<$}{<}\#number of shards\texorpdfstring{$>$}{>}} is used, the tests to execute will be split evenly in to the given number of sets, identified by indicies starting at 0. The tests in the set given by {\ttfamily -\/-\/shard-\/index \texorpdfstring{$<$}{<}\#shard index to run\texorpdfstring{$>$}{>}} will be executed. The default shard count is {\ttfamily 1}, and the default index to run is {\ttfamily 0}. It is an error to specify a shard index greater than the number of shards.

This is useful when you want to split test execution across multiple processes, as is done with \href{https://docs.bazel.build/versions/main/test-encyclopedia.html\#test-sharding}{\texttt{ Bazel test sharding}}.
