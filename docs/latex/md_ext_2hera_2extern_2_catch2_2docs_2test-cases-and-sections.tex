While Catch fully supports the traditional, x\+Unit, style of class-\/based fixtures containing test case methods this is not the preferred style.

Instead Catch provides a powerful mechanism for nesting test case sections within a test case. For a more detailed discussion see the \href{tutorial.md\#test-cases-and-sections}{\texttt{ tutorial}}.

\doxylink{class_test}{Test} cases and sections are very easy to use in practice\+:


\begin{DoxyItemize}
\item {\bfseries{TEST\+\_\+\+CASE($\ast$$\ast$ {\itshape test name} \mbox{[}, {\itshape tags} \mbox{]} $\ast$$\ast$)}}
\item {\bfseries{SECTION($\ast$$\ast$ {\itshape section name}, \mbox{[}, {\itshape section description} \mbox{]} $\ast$$\ast$)}}
\end{DoxyItemize}

{\itshape test name} and {\itshape section name} are free form, quoted, strings. The optional {\itshape tags} argument is a quoted string containing one or more tags enclosed in square brackets, and are discussed below. {\itshape section description} can be used to provide long form description of a section while keeping the {\itshape section name} short for use with the \href{command-line.md\#specify-the-section-to-run}{\texttt{ {\ttfamily -\/c} command line parameter}}.

{\bfseries{\doxylink{class_test}{Test} names must be unique within the Catch executable.}}

For examples see the \href{tutorial.md\#top}{\texttt{ Tutorial}}\hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2test-cases-and-sections_autotoc_md481}{}\doxysubsection{Tags}\label{md_ext_2hera_2extern_2_catch2_2docs_2test-cases-and-sections_autotoc_md481}
Tags allow an arbitrary number of additional strings to be associated with a test case. \doxylink{class_test}{Test} cases can be selected (for running, or just for listing) by tag -\/ or even by an expression that combines several tags. At their most basic level they provide a simple way to group several related tests together.

As an example -\/ given the following test cases\+: \begin{DoxyVerb}TEST_CASE( "A", "[widget]" ) { /* ... */ }
TEST_CASE( "B", "[widget]" ) { /* ... */ }
TEST_CASE( "C", "[gadget]" ) { /* ... */ }
TEST_CASE( "D", "[widget][gadget]" ) { /* ... */ }
\end{DoxyVerb}
 The tag expression, {\ttfamily \char`\"{}\mbox{[}widget\mbox{]}\char`\"{}} selects \doxylink{struct_a}{A}, \doxylink{struct_b}{B} \& D. {\ttfamily \char`\"{}\mbox{[}gadget\mbox{]}\char`\"{}} selects C \& D. {\ttfamily \char`\"{}\mbox{[}widget\mbox{]}\mbox{[}gadget\mbox{]}\char`\"{}} selects just D and {\ttfamily \char`\"{}\mbox{[}widget\mbox{]},\mbox{[}gadget\mbox{]}\char`\"{}} selects all four test cases.

For more detail on command line selection see \href{command-line.md\#specifying-which-tests-to-run}{\texttt{ the command line docs}}

Tag names are not case sensitive and can contain any ASCII characters. This means that tags {\ttfamily \mbox{[}tag with spaces\mbox{]}} and {\ttfamily \mbox{[}I said \char`\"{}good day\char`\"{}\mbox{]}} are both allowed tags and can be filtered on. However, escapes are not supported however and {\ttfamily \mbox{[}\textbackslash{}\mbox{]}\mbox{]}} is not a valid tag.

The same tag can be specified multiple times for a single test case, but only one of the instances of identical tags will be kept. Which one is kept is functionally random.\hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2test-cases-and-sections_autotoc_md482}{}\doxysubsubsection{Special Tags}\label{md_ext_2hera_2extern_2_catch2_2docs_2test-cases-and-sections_autotoc_md482}
All tag names beginning with non-\/alphanumeric characters are reserved by Catch. Catch defines a number of \char`\"{}special\char`\"{} tags, which have meaning to the test runner itself. These special tags all begin with a symbol character. Following is a list of currently defined special tags and their meanings.


\begin{DoxyItemize}
\item {\ttfamily \mbox{[}.\mbox{]}} -\/ causes test cases to be skipped from the default list (i.\+e. when no test cases have been explicitly selected through tag expressions or name wildcards). The hide tag is often combined with another, user, tag (for example {\ttfamily \mbox{[}.\mbox{]}\mbox{[}integration\mbox{]}} -\/ so all integration tests are excluded from the default run but can be run by passing {\ttfamily \mbox{[}integration\mbox{]}} on the command line). As a short-\/cut you can combine these by simply prefixing your user tag with a {\ttfamily .} -\/ e.\+g. {\ttfamily \mbox{[}.integration\mbox{]}}.
\item {\ttfamily \mbox{[}!throws\mbox{]}} -\/ lets Catch know that this test is likely to throw an exception even if successful. This causes the test to be excluded when running with {\ttfamily -\/e} or {\ttfamily -\/-\/nothrow}.
\item {\ttfamily \mbox{[}!mayfail\mbox{]}} -\/ doesn\textquotesingle{}t fail the test if any given assertion fails (but still reports it). This can be useful to flag a work-\/in-\/progress, or a known issue that you don\textquotesingle{}t want to immediately fix but still want to track in your tests.
\item {\ttfamily \mbox{[}!shouldfail\mbox{]}} -\/ like {\ttfamily \mbox{[}!mayfail\mbox{]}} but {\itshape fails} the test if it {\itshape passes}. This can be useful if you want to be notified of accidental, or third-\/party, fixes.
\item {\ttfamily \mbox{[}!nonportable\mbox{]}} -\/ Indicates that behaviour may vary between platforms or compilers.
\item {\ttfamily \mbox{[}\#\texorpdfstring{$<$}{<}filename\texorpdfstring{$>$}{>}\mbox{]}} -\/ running with {\ttfamily -\/\#} or {\ttfamily -\/-\/filenames-\/as-\/tags} causes Catch to add the filename, prefixed with {\ttfamily \#} (and with any extension stripped), as a tag to all contained tests, e.\+g. tests in testfile.\+cpp would all be tagged {\ttfamily \mbox{[}\#testfile\mbox{]}}.
\item {\ttfamily \mbox{[}@\texorpdfstring{$<$}{<}alias\texorpdfstring{$>$}{>}\mbox{]}} -\/ tag aliases all begin with {\ttfamily @} (see below).
\item {\ttfamily \mbox{[}!benchmark\mbox{]}} -\/ this test case is actually a benchmark. This is an experimental feature, and currently has no documentation. If you want to try it out, look at {\ttfamily projects/\+Self\+Test/\+Benchmark.\+tests.\+cpp} for details.
\end{DoxyItemize}\hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2test-cases-and-sections_autotoc_md483}{}\doxysubsection{Tag aliases}\label{md_ext_2hera_2extern_2_catch2_2docs_2test-cases-and-sections_autotoc_md483}
Between tag expressions and wildcarded test names (as well as combinations of the two) quite complex patterns can be constructed to direct which test cases are run. If a complex pattern is used often it is convenient to be able to create an alias for the expression. This can be done, in code, using the following form\+: \begin{DoxyVerb}CATCH_REGISTER_TAG_ALIAS( <alias string>, <tag expression> )
\end{DoxyVerb}
 Aliases must begin with the {\ttfamily @} character. An example of a tag alias is\+: \begin{DoxyVerb}CATCH_REGISTER_TAG_ALIAS( "[@nhf]", "[failing]~[.]" )
\end{DoxyVerb}
 Now when {\ttfamily \mbox{[}@nhf\mbox{]}} is used on the command line this matches all tests that are tagged {\ttfamily \mbox{[}failing\mbox{]}}, but which are not also hidden.\hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2test-cases-and-sections_autotoc_md484}{}\doxysubsection{BDD-\/style test cases}\label{md_ext_2hera_2extern_2_catch2_2docs_2test-cases-and-sections_autotoc_md484}
In addition to Catch\textquotesingle{}s take on the classic style of test cases, Catch supports an alternative syntax that allow tests to be written as \char`\"{}executable specifications\char`\"{} (one of the early goals of \href{http://dannorth.net/introducing-bdd/}{\texttt{ Behaviour Driven Development}}). This set of macros map on to {\ttfamily TEST\+\_\+\+CASE}s and {\ttfamily SECTION}s, with a little internal support to make them smoother to work with.


\begin{DoxyItemize}
\item {\bfseries{SCENARIO($\ast$$\ast$ {\itshape scenario name} \mbox{[}, {\itshape tags} \mbox{]} $\ast$$\ast$)}}
\end{DoxyItemize}

This macro maps onto {\ttfamily TEST\+\_\+\+CASE} and works in the same way, except that the test case name will be prefixed by \char`\"{}\+Scenario\+: \char`\"{}


\begin{DoxyItemize}
\item {\bfseries{GIVEN($\ast$$\ast$ {\itshape something} $\ast$$\ast$)}}
\item {\bfseries{WHEN($\ast$$\ast$ {\itshape something} $\ast$$\ast$)}}
\item {\bfseries{THEN($\ast$$\ast$ {\itshape something} $\ast$$\ast$)}}
\end{DoxyItemize}

These macros map onto {\ttfamily SECTION}s except that the section names are the {\itshape something} texts prefixed by \char`\"{}given\+: \char`\"{}, \char`\"{}when\+: \char`\"{} or \char`\"{}then\+: \char`\"{} respectively. These macros also map onto the AAA or \doxylink{struct_a}{A}\textsuperscript{3} test pattern (standing either for \href{http://wiki.c2.com/?AssembleActivateAssert}{\texttt{ Assemble-\/\+Activate-\/\+Assert}} or \href{http://wiki.c2.com/?ArrangeActAssert}{\texttt{ Arrange-\/\+Act-\/\+Assert}}), and in this context, the macros provide both code documentation and reporting of these parts of a test case without the need for extra comments or code to do so.

Semantically, a {\ttfamily GIVEN} clause may have multiple {\itshape independent} {\ttfamily WHEN} clauses within it. This allows a test to have, e.\+g., one set of \char`\"{}given\char`\"{} objects and multiple subtests using those objects in various ways in each of the {\ttfamily WHEN} clauses without repeating the initialisation from the {\ttfamily GIVEN} clause. When there are {\itshape dependent} clauses -- such as a second {\ttfamily WHEN} clause that should only happen {\itshape after} the previous {\ttfamily WHEN} clause has been executed and validated -- there are additional macros starting with {\ttfamily AND\+\_\+}\+:


\begin{DoxyItemize}
\item {\bfseries{AND\+\_\+\+GIVEN($\ast$$\ast$ {\itshape something} $\ast$$\ast$)}}
\item {\bfseries{AND\+\_\+\+WHEN($\ast$$\ast$ {\itshape something} $\ast$$\ast$)}}
\item {\bfseries{AND\+\_\+\+THEN($\ast$$\ast$ {\itshape something} $\ast$$\ast$)}}
\end{DoxyItemize}

These are used to chain {\ttfamily GIVEN}s, {\ttfamily WHEN}s and {\ttfamily THEN}s together. The {\ttfamily AND\+\_\+$\ast$} clause is placed {\itshape inside} the clause on which it depends. There can be multiple {\itshape independent} clauses that are all {\itshape dependent} on a single outer clause.

When any of these macros are used the console reporter recognises them and formats the test case header such that the Givens, Whens and Thens are aligned to aid readability.

Other than the additional prefixes and the formatting in the console reporter these macros behave exactly as {\ttfamily TEST\+\_\+\+CASE}s and {\ttfamily SECTION}s. As such there is nothing enforcing the correct sequencing of these macros -\/ that\textquotesingle{}s up to the programmer!\hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2test-cases-and-sections_autotoc_md485}{}\doxysubsection{Type parametrised test cases}\label{md_ext_2hera_2extern_2_catch2_2docs_2test-cases-and-sections_autotoc_md485}
In addition to {\ttfamily TEST\+\_\+\+CASE}s, Catch2 also supports test cases parametrised by types, in the form of {\ttfamily TEMPLATE\+\_\+\+TEST\+\_\+\+CASE}, {\ttfamily TEMPLATE\+\_\+\+PRODUCT\+\_\+\+TEST\+\_\+\+CASE} and {\ttfamily TEMPLATE\+\_\+\+LIST\+\_\+\+TEST\+\_\+\+CASE}.


\begin{DoxyItemize}
\item {\bfseries{TEMPLATE\+\_\+\+TEST\+\_\+\+CASE($\ast$$\ast$ {\itshape test name} , {\itshape tags}, {\itshape type1}, {\itshape type2}, ..., {\itshape typen} $\ast$$\ast$)}}
\end{DoxyItemize}

\begin{quote}
\href{https://github.com/catchorg/Catch2/issues/1437}{\texttt{ Introduced}} in Catch2 2.\+5.\+0. \end{quote}
{\itshape test name} and {\itshape tag} are exactly the same as they are in {\ttfamily TEST\+\_\+\+CASE}, with the difference that the tag string must be provided (however, it can be empty). {\itshape type1} through {\itshape typen} is the list of types for which this test case should run, and, inside the test code, the current type is available as the {\ttfamily Test\+Type} type.

Because of limitations of the C++ preprocessor, if you want to specify a type with multiple template parameters, you need to enclose it in parentheses, e.\+g. {\ttfamily std\+::map\texorpdfstring{$<$}{<}int, std\+::string\texorpdfstring{$>$}{>}} needs to be passed as {\ttfamily (std\+::map\texorpdfstring{$<$}{<}int, std\+::string\texorpdfstring{$>$}{>})}.

\item {\bfseries{TEMPLATE\+\_\+\+PRODUCT\+\_\+\+TEST\+\_\+\+CASE($\ast$$\ast$ {\itshape test name} , {\itshape tags}, ({\itshape template-\/type1}, {\itshape template-\/type2}, ..., {\itshape template-\/typen}), ({\itshape template-\/arg1}, {\itshape template-\/arg2}, ..., {\itshape template-\/argm}) $\ast$$\ast$)}}

\begin{quote}
\href{https://github.com/catchorg/Catch2/issues/1468}{\texttt{ Introduced}} in Catch2 2.\+6.\+0. \end{quote}
{\itshape template-\/type1} through {\itshape template-\/typen} is list of template template types which should be combined with each of {\itshape template-\/arg1} through {\itshape template-\/argm}, resulting in {\itshape n $\ast$ m} test cases. Inside the test case, the resulting type is available under the name of {\ttfamily Test\+Type}.

To specify more than 1 type as a single {\itshape template-\/type} or {\itshape template-\/arg}, you must enclose the types in an additional set of parentheses, e.\+g. {\ttfamily ((int, float), (char, double))} specifies 2 template-\/args, each consisting of 2 concrete types ({\ttfamily int}, {\ttfamily float} and {\ttfamily char}, {\ttfamily double} respectively). You can also omit the outer set of parentheses if you specify only one type as the full set of either the {\itshape template-\/types}, or the {\itshape template-\/args}.

\hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2test-cases-and-sections_autotoc_md486}{}\doxysubsection{Signature based parametrised test cases}\label{md_ext_2hera_2extern_2_catch2_2docs_2test-cases-and-sections_autotoc_md486}
\begin{quote}
\href{https://github.com/catchorg/Catch2/issues/1609}{\texttt{ Introduced}} in Catch2 2.\+8.\+0. \end{quote}
In addition to type parametrised test cases Catch2 also supports signature base parametrised test cases, in form of {\ttfamily TEMPLATE\+\_\+\+TEST\+\_\+\+CASE\+\_\+\+SIG} and {\ttfamily TEMPLATE\+\_\+\+PRODUCT\+\_\+\+TEST\+\_\+\+CASE\+\_\+\+SIG}. These test cases have similar syntax like type parametrised test cases, with one additional positional argument which specifies the signature.\hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2test-cases-and-sections_autotoc_md487}{}\doxysubsubsection{Signature}\label{md_ext_2hera_2extern_2_catch2_2docs_2test-cases-and-sections_autotoc_md487}
Signature has some strict rules for these tests cases to work properly\+:
\begin{DoxyItemize}
\item signature with multiple template parameters e.\+g. {\ttfamily typename T, size\+\_\+t \doxylink{struct_s}{S}} must have this format in test case declaration {\ttfamily ((typename T, size\+\_\+t \doxylink{struct_s}{S}), T, \doxylink{struct_s}{S})}
\item signature with variadic template arguments e.\+g. {\ttfamily typename T, size\+\_\+t \doxylink{struct_s}{S}, typename...Ts} must have this format in test case declaration {\ttfamily ((typename T, size\+\_\+t \doxylink{struct_s}{S}, typename...Ts), T, \doxylink{struct_s}{S}, Ts...)}
\item signature with single non type template parameter e.\+g. {\ttfamily int V} must have this format in test case declaration {\ttfamily ((int V), V)}
\item signature with single type template parameter e.\+g. {\ttfamily typename T} should not be used as it is in fact {\ttfamily TEMPLATE\+\_\+\+TEST\+\_\+\+CASE}
\end{DoxyItemize}

Currently Catch2 support up to 11 template parameters in signature\hypertarget{md_ext_2hera_2extern_2_catch2_2docs_2test-cases-and-sections_autotoc_md488}{}\doxysubsubsection{Examples}\label{md_ext_2hera_2extern_2_catch2_2docs_2test-cases-and-sections_autotoc_md488}
